# REQ-02 — เอกสารความต้องการ: การเร่งรัดการอัพเดตสถานะดำเนินการ (Status-Update Escalation)

> **ระบบ:** ระบบรับเรื่องร้องเรียนศูนย์ดำรงธรรมจังหวัดศรีสะเกษ (DCMS)
> **ขอบเขตเอกสาร:** กลไกแจ้งเตือนแบบขั้นบันได (escalation) เมื่อเรื่องที่อยู่ระหว่างดำเนินการ **ไม่มีการอัพเดตสถานะดำเนินการ**
> **สถานะ:** Requirement (อนุมัติแล้ว นำไปปรับปรุงเอกสาร Planning)
> **วันที่:** 2026-06-19

---

## 1. วัตถุประสงค์ (Purpose)

กระตุ้นให้เจ้าหน้าที่หน่วยงานอัพเดตความคืบหน้าของเรื่องร้องเรียนอย่างสม่ำเสมอ และยกระดับการแจ้งเตือนไปยังหัวหน้าหน่วยงานเมื่อเรื่องค้างไม่มีการอัพเดตเป็นเวลานาน เพื่อลดเรื่องตกหล่น/ถูกลืม

> **หมายเหตุ:** กลไกนี้เป็นคนละชุดกับการแจ้งเตือน SLA (ใกล้ครบกำหนด/เกินกำหนด) — อิงระยะเวลาที่ "ไม่มีการอัพเดตความคืบหน้า" ไม่ใช่ Due Date ของประเภทเรื่อง

---

## 2. นิยามและจุดเริ่มนับ (Definitions)

| คำ | ความหมาย |
|----|---------|
| **อัพเดตสถานะดำเนินการ** | การบันทึกความคืบหน้า (`complaint_updates` ประเภท `PROGRESS`) โดยเจ้าหน้าที่หน่วยงาน |
| **จุดเริ่มนับ (Anchor)** | เวลาอัพเดตความคืบหน้าครั้งล่าสุด — ถ้ายังไม่เคยอัพเดต ใช้ **วันที่หน่วยงานรับเรื่อง (`accepted_at`)** |
| **เจ้าหน้าที่ผู้รับผิดชอบ (ตนเอง)** | เจ้าหน้าที่หน่วยงาน (agency_officer) ที่กดรับเรื่อง (`accepted_by`) |
| **หัวหน้าหน่วยงาน** | agency_head ของหน่วยงานที่รับผิดชอบเรื่องนั้น |

---

## 3. ความต้องการเชิงฟังก์ชัน (Functional Requirements)

| ID | ความต้องการ | ลำดับความสำคัญ |
|----|-------------|:---:|
| RQ-E1 | ระบบต้องแจ้งเตือนเร่งรัดการอัพเดตสถานะ **3 ครั้ง** ตามเกณฑ์ระยะเวลาที่ไม่มีการอัพเดต | High |
| RQ-E2 | **ครั้งที่ 1** — เมื่อครบ **30 วัน** ไม่มีการอัพเดต → แจ้ง **เจ้าหน้าที่ผู้รับผิดชอบ (ตนเอง)** | High |
| RQ-E3 | **ครั้งที่ 2** — เมื่อครบ **15 วันหลังครั้งที่ 1** (รวม 45 วัน) ยังไม่มีการอัพเดต → แจ้ง **หัวหน้าหน่วยงาน + เจ้าหน้าที่ผู้รับผิดชอบ** | High |
| RQ-E4 | **ครั้งที่ 3** — เมื่อครบ **7 วันหลังครั้งที่ 2** (รวม 52 วัน) ยังไม่มีการอัพเดต → แจ้ง **หัวหน้าหน่วยงาน + เจ้าหน้าที่ผู้รับผิดชอบ** | High |
| RQ-E5 | เมื่อมีการอัพเดตความคืบหน้าใหม่ → **รีเซ็ต** วงรอบและเริ่มนับ 30 วันใหม่ | High |
| RQ-E6 | แต่ละครั้ง (ระดับ) ต้องส่ง **ครั้งเดียว** ไม่ส่งซ้ำทุกวัน | Medium |
| RQ-E7 | ใช้กับเรื่องที่อยู่ระหว่างดำเนินการที่หน่วยงาน (`ACCEPTED`/`IN_PROGRESS`) เท่านั้น ไม่รวมเรื่องที่ `RESOLVED/CLOSED/REJECTED` | Medium |

## 4. ตารางสรุปเกณฑ์ (Escalation Matrix)

| ครั้งที่ | นับจาก Anchor | สะสมรวม | ผู้รับแจ้งเตือน | Notification Type |
|:---:|:---:|:---:|---|---|
| 1 | ครบ 30 วัน | 30 วัน | เจ้าหน้าที่ผู้รับผิดชอบ (ตนเอง) | `NO_UPDATE_L1` (N-10) |
| 2 | +15 วันจากครั้งที่ 1 | 45 วัน | หัวหน้าหน่วยงาน + เจ้าหน้าที่ผู้รับผิดชอบ | `NO_UPDATE_L2` (N-11) |
| 3 | +7 วันจากครั้งที่ 2 | 52 วัน | หัวหน้าหน่วยงาน + เจ้าหน้าที่ผู้รับผิดชอบ | `NO_UPDATE_L3` (N-12) |

## 5. ข้อมูลที่ระบบต้องจัดเก็บ (Data Fields)

เพิ่มในตาราง `complaints`:

| ฟิลด์ | ชนิด | คำอธิบาย |
|------|------|---------|
| `last_progress_at` | DATETIME | เวลาอัพเดตความคืบหน้าล่าสุด (หรือ `accepted_at` ถ้ายังไม่เคยอัพเดต) — Anchor |
| `escalation_level` | TINYINT (0–3) | ระดับการเร่งรัดที่ส่งไปแล้ว (0=ยังไม่ส่ง, 1, 2, 3) — กันส่งซ้ำ |
| `last_escalation_at` | DATETIME | เวลาส่งแจ้งเตือนเร่งรัดครั้งล่าสุด |

---

## 6. กติกาทางธุรกิจ (Business Rules)

| # | กติกา |
|---|------|
| BR-E1 | Anchor = อัพเดตล่าสุด (`PROGRESS`); ถ้ายังไม่เคยอัพเดต = `accepted_at` |
| BR-E2 | เกณฑ์สะสม 30 → 45 → 52 วัน (นับเป็นวันตามปฏิทิน) |
| BR-E3 | ส่งครั้งเดียวต่อระดับ โดยอ้างอิง `escalation_level` (L1 เมื่อ level<1, L2 เมื่อ level<2, L3 เมื่อ level<3) |
| BR-E4 | มีอัพเดตความคืบหน้า → `last_progress_at = NOW()`, `escalation_level = 0` |
| BR-E5 | ขอบเขต: เฉพาะสถานะ `ACCEPTED`/`IN_PROGRESS` |
| BR-E6 | ตรวจและส่งโดย Scheduled Job (node-cron) ทุกวัน 08:00 |
| BR-E7 | เป็นกลไกแยกจาก SLA near-due/overdue (ไม่อิง `due_date`/`is_overdue`) |

## 7. ตรรกะการทำงาน (Pseudo Logic)

```
งาน check_no_update_escalation — รันทุกวัน 08:00
สำหรับแต่ละ complaint ที่ status IN ('ACCEPTED','IN_PROGRESS'):
    days = DATEDIFF(NOW(), last_progress_at)
    ถ้า days >= 52 และ escalation_level < 3:
        แจ้ง NO_UPDATE_L3 → หัวหน้าหน่วยงาน + เจ้าหน้าที่ผู้รับผิดชอบ ; escalation_level = 3
    ไม่งั้นถ้า days >= 45 และ escalation_level < 2:
        แจ้ง NO_UPDATE_L2 → หัวหน้าหน่วยงาน + เจ้าหน้าที่ผู้รับผิดชอบ ; escalation_level = 2
    ไม่งั้นถ้า days >= 30 และ escalation_level < 1:
        แจ้ง NO_UPDATE_L1 → เจ้าหน้าที่ผู้รับผิดชอบ ; escalation_level = 1
    set last_escalation_at = NOW() (เมื่อมีการส่ง)

เมื่อรับ POST /api/complaints/:id/updates (PROGRESS):
    last_progress_at = NOW() ; escalation_level = 0
```

## 8. เกณฑ์การยอมรับ (Acceptance Criteria)

- [ ] ✅ เรื่อง IN_PROGRESS ที่ไม่มีอัพเดตครบ 30 วัน → เจ้าหน้าที่ผู้รับผิดชอบได้รับแจ้งเตือน `NO_UPDATE_L1`
- [ ] ✅ ครบ 45 วันไม่มีอัพเดต → หัวหน้าหน่วยงาน **และ** เจ้าหน้าที่ผู้รับผิดชอบได้รับ `NO_UPDATE_L2`
- [ ] ✅ ครบ 52 วันไม่มีอัพเดต → หัวหน้าหน่วยงาน **และ** เจ้าหน้าที่ผู้รับผิดชอบได้รับ `NO_UPDATE_L3`
- [ ] ⛔ แต่ละระดับส่ง **ครั้งเดียว** — รัน cron ซ้ำวันถัดไปต้องไม่ส่งระดับเดิมซ้ำ
- [ ] ✅ มีการอัพเดตความคืบหน้า → `escalation_level` กลับเป็น 0 และเริ่มนับ 30 วันใหม่ (ไม่ส่ง L1 จนกว่าจะครบ 30 วันอีกครั้ง)
- [ ] ⛔ เรื่องที่ `RESOLVED/CLOSED/REJECTED` ไม่ถูกส่งแจ้งเตือนเร่งรัด
- [ ] ✅ เรื่องที่ยังไม่เคยอัพเดตเลย นับจาก `accepted_at` ได้ถูกต้อง
