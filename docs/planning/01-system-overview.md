## 2. System Overview

ระบบเว็บแอปพลิเคชันสำหรับ**ศูนย์ดำรงธรรมจังหวัด** เพื่อบริหารจัดการเรื่องร้องเรียน/ร้องทุกข์ของประชาชนอย่างเป็นระบบ ครอบคลุมตั้งแต่:

1. **รับเรื่อง** — บันทึกข้อมูลร้องเรียนจากหลายช่องทาง (จดหมาย, ลงพื้นที่/หน่วยเคลื่อนที่เร็ว, ตู้ราชสีห์, สายด่วน 1567, Walk In, เว็บไซต์, แอปฯ MOI1567, ผ่านหน่วยงานอื่น, อื่น ๆ)
2. **คัดกรองและจัดประเภท** — ตรวจสอบข้อมูล จำแนกประเภทเรื่อง กำหนดพื้นที่
3. **ส่งต่อหน่วยงาน** — มอบหมายเรื่องไปยังหน่วยงานที่รับผิดชอบพร้อมกำหนดระยะเวลา (SLA)
4. **ติดตามความคืบหน้า** — หน่วยงานอัปเดตผลดำเนินการ ศูนย์ดำรงธรรมติดตามและเร่งรัด
5. **แจ้งผลและปิดเรื่อง** — แจ้งผลให้ประชาชน ปิดเรื่องเมื่อดำเนินการเสร็จสิ้น
6. **รายงานและ Dashboard** — แสดงภาพรวมสถิติ KPI สำหรับผู้บริหาร

### Tech Stack

| Layer | Technology |
|-------|-----------|
| Frontend | React + Vite + MUI (Material UI) |
| Backend | Node.js + Express |
| Database | MySQL 8 |
| Container | Docker Compose (Dev) |
| DB Admin | phpMyAdmin (Dev) |
| Deploy | Railway (Single-container Dockerfile) |

## 3. Objectives

| # | วัตถุประสงค์ |
|---|-------------|
| O-1 | ให้มีระบบรับเรื่องร้องเรียนแบบดิจิทัลที่สามารถบันทึกข้อมูลได้ครบถ้วนและเป็นมาตรฐาน |

| O-7 | สามารถ Export รายงานเพื่อใช้ในการประชุมหรือส่งต่อผู้บังคับบัญชา |

## 4. Problems of Current Process

| # | ปัญหาของระบบ/กระบวนการเดิม |
|---|---------------------------|
| P-1 | ใช้สมุดบันทึกหรือ Excel ในการจดข้อมูล ทำให้ค้นหาและติดตามยาก |
| P-2 | ไม่มีระบบติดตามสถานะ — ไม่รู้ว่าเรื่องไหนอยู่ระหว่างดำเนินการ เสร็จแล้ว หรือค้างนานแค่ไหน |
| P-3 | การส่งต่อเรื่องไปยังหน่วยงานทำผ่านหนังสือราชการหรือวาจา ไม่มี Tracking |
| P-4 | ไม่สามารถดูสถิติภาพรวมเรื่องร้องเรียนแบบ Real-time ได้ ต้องนับมือ |
| P-5 | ประชาชนไม่ทราบความคืบหน้าของเรื่องที่ร้องเรียนไป ต้องโทรถาม |
| P-6 | เรื่องร้องเรียนตกหล่นหรือถูกลืม เพราะไม่มีระบบเตือนอัตโนมัติ |
| P-7 | ข้อมูลกระจัดกระจาย ทำรายงานประจำเดือน/ประจำปีลำบาก |
| P-8 | ไม่มี Audit Trail — ไม่สามารถตรวจสอบย้อนหลังได้ว่าใครทำอะไรเมื่อไหร่ |

## 7. System Boundary

### ผู้เกี่ยวข้อง (Actors)

```
ประชาชน ──────► [หน้าเว็บสาธารณะ: ยื่นเรื่อง / ติดตามสถานะ]

                              ▼
```

- **ภายในระบบ**: การรับเรื่อง, คัดกรอง, ส่งต่อ, ติดตาม, แจ้งเตือน, Dashboard, Report, User Management, Master Data

## 12. Key Decisions

| # | การตัดสินใจ | เหตุผล |
|---|-----------|--------|
| KD-1 | ใช้ **React + Vite + MUI** สำหรับ Frontend | MUI มี Component สำเร็จรูปเหมาะกับระบบ Back-office, Vite เร็วกว่า CRA |
| KD-2 | ใช้ **Node.js + Express** สำหรับ Backend | ใช้ JavaScript ตลอด Full-stack, เหมาะกับทีมที่คุ้นเคย Node.js |
| KD-3 | ใช้ **MySQL 8** สำหรับ Database | เป็น RDBMS มาตรฐาน รองรับ Transaction, Foreign Key, และภาษาไทย (utf8mb4) |
| KD-4 | ใช้ **Docker Compose** สำหรับ Development และ Production | ทำให้ Environment เหมือนกัน ทุกคน Setup ได้ง่าย |
| KD-5 | ใช้ **JWT** สำหรับ Authentication | Stateless, เหมาะกับ REST API |
| KD-6 | ใช้ **Role-Based Access Control (RBAC)** | เหมาะกับระบบราชการที่มีลำดับชั้นชัดเจน |
| KD-7 | หน้าสาธารณะ (ยื่นเรื่อง/ติดตามสถานะ) **ไม่ต้อง Login** (Guest) | เพื่อให้ประชาชนเข้าถึงง่าย |
| KD-8 | แจ้งเตือนใช้ **In-app Notification** ก่อน | ยังไม่เชื่อม SMS/Email ใน Phase แรก |
| KD-9 | เพิ่ม **ระบบสมาชิกประชาชน** (Auth แยกจาก staff) ควบคู่ Guest | ให้ติดตามเรื่อง/แก้ข้อมูลได้ โดยไม่ปน RBAC เจ้าหน้าที่ (ดู `03` §12) |
| KD-10 | รองรับ **ปกปิดตัวตน** โดย mask ที่ Backend ทุก Role + เปิดเผยเฉพาะ super_admin | คุ้มครองผู้ร้องตาม PDPA + ตรวจสอบย้อนหลังได้ |

## สรุป

ระบบ DCMS จะเป็นเว็บแอปพลิเคชันที่ช่วยให้ศูนย์ดำรงธรรมจังหวัดสามารถ:

- **รับเรื่อง** ร้องเรียนจากประชาชนผ่านหลายช่องทาง
- **บริหารจัดการ** เรื่องร้องเรียนอย่างเป็นระบบด้วย Workflow ที่ชัดเจน
- **ส่งต่อและติดตาม** เรื่องไปยังหน่วยงานที่รับผิดชอบพร้อม SLA
- **แสดงผลภาพรวม** ผ่าน Dashboard สำหรับผู้บริหาร
- **สร้างรายงาน** เพื่อใช้ในการประชุมและตัดสินใจ
