## 1. วัตถุประสงค์ของระบบ

| # | วัตถุประสงค์ | ตัวชี้วัดความสำเร็จ |
|---|-------------|-------------------|
| O-1 | เปลี่ยนกระบวนการรับเรื่องร้องเรียนจากกระดาษ/Excel ให้เป็นระบบดิจิทัล | เจ้าหน้าที่ทุกคนใช้ระบบแทนสมุดบันทึก |
| O-2 | ให้ทุกเรื่องร้องเรียนมีสถานะที่ติดตามได้ตลอดเวลา | 100% ของเรื่องมีสถานะที่ถูกบันทึกในระบบ |
| O-3 | ลดเรื่องตกหล่นและเรื่องที่ถูกลืม | เรื่องเกินกำหนด (Overdue) ลดลง ≥ 50% |
| O-4 | ผู้บริหารสามารถดูภาพรวมสถิติแบบ Real-time | มี Dashboard ที่แสดงข้อมูลอัปเดตทันที |
| O-5 | ประชาชนสามารถติดตามสถานะเรื่องร้องเรียนได้ด้วยตนเอง | มีหน้าเว็บสาธารณะสำหรับตรวจสอบสถานะ |
| O-6 | มีข้อมูลสถิติสำหรับจัดทำรายงานประจำเดือน/ปี | Export รายงานได้ภายใน 1 นาที |
| O-7 | มีประวัติการดำเนินการ (Audit Trail) ที่ตรวจสอบย้อนหลังได้ | ทุกการเปลี่ยนแปลงถูกบันทึก |

## 2. ปัญหาของระบบเดิม

| # | ปัญหา | ผลกระทบ | ความรุนแรง |
|---|-------|---------|-----------|
| P-1 | ใช้สมุดบันทึกหรือ Excel จดข้อมูลเรื่องร้องเรียน | ค้นหาข้อมูลยาก ข้อมูลสูญหายง่าย | สูง |
| P-2 | ไม่มีระบบติดตามสถานะเรื่องร้องเรียน | ไม่รู้ว่าเรื่องไหนอยู่ขั้นตอนไหน ใครรับผิดชอบ | สูง |
| P-3 | การส่งต่อเรื่องใช้หนังสือราชการหรือวาจา | ช้า ไม่มีหลักฐาน ติดตามไม่ได้ | สูง |
| P-4 | เรื่องตกหล่นหรือถูกลืม ไม่มีการเตือน | ประชาชนรอนาน เสียความเชื่อมั่น | สูง |
| P-5 | ทำรายงานสรุปยาก ต้องนับมือ | ใช้เวลามาก ข้อมูลไม่แม่นยำ | ปานกลาง |
| P-6 | ประชาชนต้องโทรถามความคืบหน้า | เพิ่มภาระเจ้าหน้าที่ ประชาชนไม่สะดวก | ปานกลาง |
| P-7 | ข้อมูลกระจัดกระจายหลายที่ | ไม่สามารถมองภาพรวมได้ ข้อมูลไม่สอดคล้องกัน | ปานกลาง |
| P-8 | ไม่มี Audit Trail | ไม่สามารถตรวจสอบย้อนหลังได้ว่าใครทำอะไรเมื่อไหร่ | ปานกลาง |
| P-9 | ไม่มีการกำหนด SLA ที่ชัดเจนในระบบ | ไม่รู้ว่าเรื่องไหนเกินกำหนด ไม่สามารถเร่งรัดได้ทันเวลา | ปานกลาง |

## 3. เป้าหมายของระบบใหม่

| # | เป้าหมาย | รายละเอียด |
|---|---------|-----------|
| G-1 | **ศูนย์กลางข้อมูล** | รวมข้อมูลเรื่องร้องเรียนทั้งหมดไว้ในที่เดียว สามารถค้นหา กรอง และเข้าถึงได้ทันที |
| G-2 | **ติดตามได้ตลอดสาย** | ทุกเรื่องมีสถานะชัดเจน ตั้งแต่รับเรื่อง → คัดกรอง → ส่งต่อ → ดำเนินการ → ปิดเรื่อง |
| G-3 | **แจ้งเตือนอัตโนมัติ** | ระบบแจ้งเตือนเมื่อมีเรื่องใหม่ ใกล้ครบกำหนด เกินกำหนด หรือมีการเปลี่ยนสถานะ |
| G-4 | **ลดภาระงานเอกสาร** | ลดการพิมพ์หนังสือราชการ ส่งต่อเรื่องผ่านระบบแทน |
| G-5 | **ข้อมูลเชิงสถิติ** | มี Dashboard และรายงานที่พร้อมใช้สำหรับการประชุมและการตัดสินใจ |
| G-6 | **ประชาชนเข้าถึงง่าย** | ประชาชนสามารถยื่นเรื่องและติดตามสถานะผ่านเว็บได้ โดยไม่ต้องเดินทาง |
| G-7 | **ตรวจสอบย้อนหลังได้** | มี Audit Log บันทึกทุกการกระทำ เพื่อความโปร่งใสและตรวจสอบได้ |
| G-8 | **รองรับหลายช่องทาง** | บันทึกเรื่องร้องเรียนได้จากทุกช่องทาง: จดหมาย, ลงพื้นที่/หน่วยเคลื่อนที่เร็ว, ตู้ราชสีห์, สายด่วน 1567, Walk In, เว็บไซต์, แอปฯ MOI1567, ผ่านหน่วยงานอื่น, อื่น ๆ |

## 4. Functional Requirements

### 4.1 การรับเรื่องร้องเรียน

| ID | Requirement | Priority | หมายเหตุ |
|----|------------|----------|---------|
| FR-001 | เจ้าหน้าที่สามารถบันทึกเรื่องร้องเรียนใหม่ได้ (ชื่อผู้ร้อง, เรื่อง, รายละเอียด, ที่อยู่, เบอร์โทร) | High | รับจากทุกช่องทาง |
| FR-002 | ระบบบันทึกช่องทางที่รับเรื่อง (จดหมาย, ลงพื้นที่/หน่วยเคลื่อนที่เร็ว, ตู้ราชสีห์, สายด่วน 1567, Walk In, เว็บไซต์, แอปฯ MOI1567, ผ่านหน่วยงานอื่น, อื่น ๆ) | High | เลือกจาก Master Data — เจ้าหน้าที่เลือกช่องทางที่รับเรื่องมา (ดู `05` §8.2) |
| FR-003 | สามารถแนบไฟล์หลักฐานประกอบเรื่องร้องเรียน (รูปภาพ, PDF, เอกสาร) ได้หลายไฟล์ ทั้งฝั่งเจ้าหน้าที่และประชาชน | High | ≤ 10 MB/ไฟล์ · jpg, jpeg, png, pdf, doc, docx (BR-11) |
| FR-004 | ระบบสร้างเลขที่เรื่องร้องเรียนอัตโนมัติ (Running Number) | High | รูปแบบ: DC-YYYYMM-XXXX |
| FR-005 | ประชาชนสามารถยื่นเรื่องร้องเรียนผ่านหน้าเว็บสาธารณะได้ **แบบไม่สมัครสมาชิก (Guest)** | Medium | หน้า Public |
| FR-006 | สามารถยื่นเรื่องแบบ **ปกปิดตัวตน (Anonymous)** ได้ — ข้อมูลผู้ร้องยังบันทึกใน DB แต่ **ไม่แสดงต่อเจ้าหน้าที่ทุก Role** | Medium | เลือกได้ทุกช่องทางการยื่น (Guest/สมาชิก/staff) — ดู `03` §5.1.1 |
| FR-007 | สามารถระบุพื้นที่/จุดเกิดเหตุ: **จังหวัด, อำเภอ, ตำบล, รหัสไปรษณีย์, สถานที่เกิดเหตุ** | High | จังหวัด/อำเภอ/ตำบล เลือกจาก Master Data |
| FR-007.1 | สามารถ **ปักหมุดจุดเกิดเหตุบนแผนที่** ได้ทั้งจาก GPS ของอุปกรณ์ หรือเลือกพิกัดเองบนแผนที่ (Leaflet) | Medium | เก็บ `latitude`/`longitude` ใน `complaints` |
| FR-007.2 | ระบบบันทึก **ชื่อเรื่องร้องเรียน** (หัวเรื่อง) | High | `title` |
| FR-007.3 | ระบบบันทึก **ประเภทงานบริการ** (ทั่วไป/สำคัญ/บัตรสนเท่ห์/ผลกระทบวงกว้าง/นโยบายสำคัญ) | High | เลือกจาก Master Data `service_types` |
| FR-007.4 | ระบบบันทึก **ลักษณะเรื่อง** (8 ลักษณะ เช่น เรื่องร้องเรียนร้องทุกข์, บริการให้คำปรึกษา, ข้อเสนอแนะ ฯลฯ) | High | เลือกจาก Master Data `complaint_natures` |
| FR-007.5 | ระบบบันทึก **ประเภทผู้ร้องเรียน** (บุคคลธรรมดา/นิติบุคคล) | High | เลือกจาก Master Data `complainant_types` |
| FR-007.6 | **บังคับกรอกเบอร์โทรศัพท์ผู้ร้อง** ทั้งกรณีออกนามและปกปิดตัวตน | High | เพื่อให้ติดต่อกลับได้ (BR-16) — เมื่อปกปิดตัวตน เบอร์ยังถูก mask จาก staff |
| FR-008 | ประชาชนสามารถยื่นเรื่อง **แบบสมาชิก (Login)** โดยระบบผูกเรื่องกับบัญชีและ pre-fill ข้อมูลจากโปรไฟล์ | Medium | `POST /api/citizen/complaints` |
| FR-009 | `super_admin` สามารถ **เปิดเผยตัวตน** ผู้ร้องของเรื่อง Anonymous ได้เฉพาะกรณีจำเป็น โดยต้องระบุเหตุผลและระบบบันทึก audit | Medium | ดู `03` §5.1.1 |
| FR-009.1 | **เจ้าหน้าที่ศูนย์ดำรงธรรมสามารถบันทึก/แจ้งเรื่องร้องทุกข์แทนประชาชนได้** พร้อมเลือกช่องทางที่รับเรื่องมา (จดหมาย, ลงพื้นที่, ตู้ราชสีห์, 1567, Walk In, MOI1567, หน่วยงานอื่น, อื่น ๆ) | High | `source = STAFF`, `POST /api/complaints` |

## 8. เงื่อนไขทางธุรกิจของระบบ (Business Rules)

| # | เงื่อนไข | รายละเอียด |
|---|---------|-----------|
| BR-01 | **เลขที่เรื่อง** | สร้างอัตโนมัติตามรูปแบบ `DC-YYYYMM-XXXX` โดย XXXX เป็น Running Number รีเซ็ตทุกเดือน |
| BR-02 | **SLA** | ระยะเวลาดำเนินการคำนวณจากวันที่ส่งต่อหน่วยงาน (ไม่ใช่วันที่รับเรื่อง) กำหนดตาม Category |
| BR-03 | **สถานะเกินกำหนด** | เรื่องที่เกิน Due Date และยังไม่ปิด ระบบแสดงเป็นสถานะ "เกินกำหนด" โดยอัตโนมัติ |
| BR-04 | **การปิดเรื่อง** | เฉพาะเจ้าหน้าที่ศูนย์ดำรงธรรมเท่านั้นที่สามารถปิดเรื่องได้ หน่วยงานทำได้เพียงส่งผลกลับ |
| BR-05 | **การส่งต่อ** | 1 เรื่องสามารถส่งต่อไปยังหลายหน่วยงานได้ แต่ต้องส่งทีละหน่วย |
| BR-06 | **การส่งผิดหน่วยงาน** | หน่วยงานสามารถ "ส่งคืน" พร้อมเหตุผล เจ้าหน้าที่ศูนย์จะส่งต่อใหม่ |
| BR-07 | **ข้อมูลส่วนบุคคล** | เลขบัตรประชาชน ชื่อ ที่อยู่ผู้ร้อง เข้าถึงได้เฉพาะเจ้าหน้าที่ศูนย์ หัวหน้าศูนย์ และ Admin |
| BR-08 | **ผู้ใช้ที่ Inactive** | ผู้ใช้ที่ถูก Deactivate ไม่สามารถ Login ได้ แต่ข้อมูลยังคงอยู่ (Soft Delete) |
| BR-09 | **Audit Log** | ทุกการเปลี่ยนสถานะเรื่องร้องเรียนต้องบันทึก: ใคร เมื่อไหร่ จากสถานะอะไรเป็นอะไร |
| BR-10 | **Public Tracking** | ประชาชนสามารถดูสถานะเรื่องได้ แต่จะเห็นเฉพาะสถานะกว้างๆ (เช่น "กำลังดำเนินการ") ไม่เห็นรายละเอียดภายใน |
| BR-11 | **ไฟล์แนบ** | จำกัดขนาดไฟล์ ≤ 10 MB ต่อไฟล์ รองรับนามสกุล: jpg, jpeg, png, pdf, doc, docx |
| BR-12 | **การลบเรื่อง** | ไม่สามารถลบเรื่องร้องเรียนได้ (Soft Delete เท่านั้น) เพื่อความโปร่งใส |
| BR-13 | **ปกปิดตัวตน (Anonymous)** | เมื่อเลือกปกปิดตัวตน ข้อมูลผู้ร้อง (และ `citizen_id`) ยังบันทึกใน DB แต่ต้องถูก mask จากเจ้าหน้าที่ **ทุก Role** ที่ระดับ Backend; เปิดเผยได้เฉพาะ super_admin พร้อมเหตุผล + บันทึก `anonymous_reveal_logs` |
| BR-14 | **บัญชีสมาชิก (Citizen)** | บัญชีสมาชิกประชาชนแยกจากตาราง `users`/RBAC ของเจ้าหน้าที่; Token ของสมาชิกเข้าถึงได้เฉพาะ `/api/citizen/*`; เห็นเฉพาะเรื่องของตนเอง |
| BR-15 | **Snapshot ข้อมูลผู้ร้อง** | ข้อมูลผู้ร้องในเรื่องร้องเรียนเก็บเป็น snapshot ณ เวลายื่น ไม่เปลี่ยนตามการแก้โปรไฟล์สมาชิกภายหลัง |
| BR-16 | **เบอร์โทรบังคับกรอก** | ทุกการยื่นเรื่อง (ออกนาม/ปกปิดตัวตน, Guest/สมาชิก) **ต้องกรอกเบอร์โทรศัพท์ผู้ร้อง** เพื่อให้ติดต่อกลับได้ — เมื่อปกปิดตัวตน เบอร์ยังถูก mask จาก staff เช่นเดียวกับข้อมูลส่วนบุคคลอื่น |
| BR-17 | **จุดเกิดเหตุ + พิกัด** | ระบบรองรับการระบุจังหวัด/อำเภอ/ตำบล/รหัสไปรษณีย์/สถานที่เกิดเหตุ และปักหมุดพิกัดบนแผนที่ (GPS หรือเลือกเอง) — พิกัดเป็น Optional |
| BR-18 | **ช่องทางร้องทุกข์ของเจ้าหน้าที่** | เจ้าหน้าที่ศูนย์บันทึกเรื่องแทนประชาชนได้ พร้อมระบุช่องทางที่รับเรื่องมา (จดหมาย, ลงพื้นที่/หน่วยเคลื่อนที่เร็ว, ตู้ราชสีห์, 1567, Walk In, MOI1567, ผ่านหน่วยงานอื่น, อื่น ๆ) |

## 9. ข้อจำกัดและข้อควรระวัง

| # | ข้อจำกัด/ข้อควรระวัง | ผลกระทบ | การรับมือ |
|---|---------------------|---------|----------|
| C-1 | **ไม่มี SMS/Email Notification** ใน Phase แรก | ประชาชนต้องเข้ามาเช็คสถานะเอง | เพิ่มใน Phase ถัดไป |
| C-2 | **ไม่เชื่อมต่อระบบกลาง** ของกระทรวงมหาดไทย | ต้อง Key ข้อมูลซ้ำถ้ามีเรื่องจากระบบกลาง | พัฒนา API Integration ภายหลัง |
| C-3 | **ข้อมูลส่วนบุคคล (PDPA)** | ต้องระวังการเก็บ/แสดง ชื่อ เลขบัตร ที่อยู่ | จำกัดสิทธิ์ตาม Role + Encrypt ข้อมูลสำคัญ |
| C-4 | **Internet ของหน่วยงาน** | บางหน่วยงานอาจมี Internet ไม่เสถียร | ออกแบบ UI ให้ทำงานได้แม้ Network ช้า |
| C-5 | **ความคุ้นเคยของเจ้าหน้าที่** | เจ้าหน้าที่บางคนอาจไม่คุ้นเคยกับระบบดิจิทัล | ออกแบบ UI ให้ง่าย + จัดอบรม |
| C-6 | **ขนาดไฟล์แนบ** | ไฟล์ขนาดใหญ่อาจกระทบ Storage และ Performance | จำกัดขนาด 10 MB/ไฟล์ |
| C-7 | **Running Number** | ถ้า Deploy หลาย Instance อาจมีปัญหา Duplicate | ใช้ Database Sequence / Lock |
| C-8 | **เวลาเซิร์ฟเวอร์** | ต้องใช้ Timezone ของประเทศไทย (Asia/Bangkok) | ตั้งค่า Timezone ที่ Server + Database |
