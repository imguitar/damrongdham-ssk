## 1. Role Overview

ระบบแบ่งผู้ใช้งานออกเป็น **8 Roles** โดย 6 Role ตามที่กำหนดไว้ และเสนอเพิ่มอีก 2 Role:

| # | Role Code | ชื่อ Role (ไทย) | ชื่อ Role (อังกฤษ) | ต้อง Login | หมายเหตุ |
|---|-----------|-----------------|-------------------|-----------|---------|
| 1 | `super_admin` | Super Admin | Super Admin | ✅ | ตามที่กำหนด |
| 2 | `admin` | ผู้ดูแลระบบ | Admin | ✅ | **เสนอเพิ่ม** |
| 3 | `officer` | เจ้าหน้าที่ศูนย์ดำรงธรรม | Damrongdham Officer | ✅ | ตามที่กำหนด |
| 4 | `chief` | หัวหน้าศูนย์ดำรงธรรม | Damrongdham Chief | ✅ | ตามที่กำหนด |
| 5 | `agency_officer` | เจ้าหน้าที่หน่วยงาน | Agency Officer | ✅ | ตามที่กำหนด (แยกย่อย) |
| 6 | `agency_head` | หัวหน้าหน่วยงาน | Agency Head | ✅ | **เสนอเพิ่ม** |
| 7 | `executive` | ผู้บริหารจังหวัด | Executive | ✅ | ตามที่กำหนด |
| 8 | `public` | ประชาชนผู้ร้องเรียน | Public | ❌ | ไม่ต้อง Login |

### เหตุผลที่เสนอ Role เพิ่มเติม

**Admin (ผู้ดูแลระบบ)**
- แยกออกจาก Super Admin เพื่อให้มีชั้นการจัดการ
- Admin ดูแล Master Data และผู้ใช้งานทั่วไป แต่ไม่สามารถจัดการ Super Admin หรือเข้าถึง System Config ระดับลึกได้
- ป้องกันความเสี่ยงจากการมี Super Admin มากเกินไป

**Agency Head (หัวหน้าหน่วยงาน)**
- แยกจาก Agency Officer เพื่อให้หัวหน้าหน่วยงานสามารถดูสรุปภาพรวมเรื่องทั้งหมดของหน่วยงาน
- ไม่จำเป็นต้องอัปเดตผลเอง แต่ดูสถิติและตรวจสอบได้

## 2. Role Description

### Role 1: Super Admin (`super_admin`)

| หัวข้อ | รายละเอียด |
|--------|-----------|
| **คำอธิบาย** | ผู้ดูแลระบบสูงสุด มีสิทธิ์ทุกอย่างในระบบ |
| **สิทธิ์ที่ทำได้** | จัดการผู้ใช้ทุก Role, จัดการ Master Data ทั้งหมด, ดูเรื่องร้องเรียนทั้งหมด, ดู Audit Log, ตั้งค่าระบบ, Export รายงาน |
| **หน้าจอที่เข้าถึง** | ทุกหน้าจอ |
| **ข้อมูลที่มองเห็น** | ทุกข้อมูลรวมถึงข้อมูลส่วนบุคคลผู้ร้อง |
| **ข้อจำกัด** | ไม่มี |
| **จำนวนที่แนะนำ** | 1–2 คน |

### Role 2: Admin (`admin`)

| หัวข้อ | รายละเอียด |
|--------|-----------|
| **คำอธิบาย** | ผู้ดูแลระบบทั่วไป จัดการผู้ใช้งานและ Master Data |
| **สิทธิ์ที่ทำได้** | CRUD ผู้ใช้ (ยกเว้น Super Admin), CRUD Master Data ทั้งหมด, ดูเรื่องร้องเรียนทั้งหมด, ดู Audit Log, Export รายงาน |
| **หน้าจอที่เข้าถึง** | ทุกหน้าจอ ยกเว้น System Config |
| **ข้อมูลที่มองเห็น** | ทุกข้อมูลรวมถึงข้อมูลส่วนบุคคลผู้ร้อง |
| **ข้อจำกัด** | ไม่สามารถสร้าง/แก้ไข/ลบ Super Admin ได้, ไม่สามารถเข้าถึง System Config |

### Role 3: เจ้าหน้าที่ศูนย์ดำรงธรรม (`officer`)

| หัวข้อ | รายละเอียด |
|--------|-----------|
| **คำอธิบาย** | เจ้าหน้าที่ปฏิบัติงานหลักของศูนย์ดำรงธรรม รับเรื่อง คัดกรอง ส่งต่อ ติดตาม ปิดเรื่อง |
| **สิทธิ์ที่ทำได้** | สร้างเรื่องร้องเรียนใหม่, แก้ไขเรื่องร้องเรียน, จัดประเภท/กำหนดพื้นที่, ส่งต่อหน่วยงาน, ติดตามเร่งรัด, ตรวจสอบผลดำเนินการ, ปิดเรื่อง, ดู Dashboard, Export รายงาน |
| **หน้าจอที่เข้าถึง** | Dashboard, Complaints (ทุก Sub-page), Notifications, Reports |
| **ข้อมูลที่มองเห็น** | เรื่องร้องเรียนทั้งหมด, ข้อมูลส่วนบุคคลผู้ร้อง, ผลดำเนินการจากหน่วยงาน |
| **ข้อจำกัด** | ไม่สามารถจัดการ Master Data หรือผู้ใช้งาน, ไม่เห็น Audit Log |

### Role 4: หัวหน้าศูนย์ดำรงธรรม (`chief`)

| หัวข้อ | รายละเอียด |
|--------|-----------|
| **คำอธิบาย** | หัวหน้าศูนย์ มีสิทธิ์เหมือนเจ้าหน้าที่ เพิ่มเติมดู Dashboard สรุปภาพรวมและอนุมัติ |
| **สิทธิ์ที่ทำได้** | ทุกอย่างที่ Officer ทำได้ + ดู Dashboard ระดับสูง, ดูรายงานสรุป, ดูเรื่องเกินกำหนด, อนุมัติ/ปฏิเสธเรื่อง |
| **หน้าจอที่เข้าถึง** | Dashboard (Full), Complaints (ทุก Sub-page), Reports (Full), Notifications |
| **ข้อมูลที่มองเห็น** | เรื่องร้องเรียนทั้งหมด, ข้อมูลส่วนบุคคลผู้ร้อง, สถิติภาพรวม |
| **ข้อจำกัด** | ไม่สามารถจัดการ Master Data หรือผู้ใช้งาน |

### Role 5: เจ้าหน้าที่หน่วยงาน (`agency_officer`)

| หัวข้อ | รายละเอียด |
|--------|-----------|
| **คำอธิบาย** | เจ้าหน้าที่ของหน่วยงานที่ได้รับมอบหมายเรื่องร้องเรียน รับเรื่องและอัปเดตผลดำเนินการ |
| **สิทธิ์ที่ทำได้** | ดูเรื่องที่ถูกส่งมาให้หน่วยงานของตน, รับเรื่อง (Accept), อัปเดตความคืบหน้า, แนบไฟล์ผลดำเนินการ, ส่งผลกลับ, ส่งคืนเรื่อง (กรณีส่งผิด) |
| **หน้าจอที่เข้าถึง** | Dashboard (เฉพาะหน่วยงาน), Assigned Complaints, Notifications |
| **ข้อมูลที่มองเห็น** | เฉพาะเรื่องที่ส่งมาให้หน่วยงานของตนเท่านั้น, **ไม่เห็น**ข้อมูลส่วนบุคคลผู้ร้อง (เลขบัตร, ที่อยู่) |
| **ข้อจำกัด** | เห็นเฉพาะเรื่องของหน่วยงานตนเอง, ไม่สามารถสร้างเรื่องใหม่, ไม่สามารถปิดเรื่อง, ไม่เห็นข้อมูลส่วนบุคคลผู้ร้อง |

### Role 6: หัวหน้าหน่วยงาน (`agency_head`)

| หัวข้อ | รายละเอียด |
|--------|-----------|
| **คำอธิบาย** | หัวหน้าของหน่วยงานที่ได้รับมอบหมาย ดูสรุปภาพรวมเรื่องของหน่วยงาน |
| **สิทธิ์ที่ทำได้** | ทุกอย่างที่ Agency Officer ทำได้ + ดูสรุปเรื่องทั้งหมดของหน่วยงาน, ดู Dashboard ระดับหน่วยงาน |
| **หน้าจอที่เข้าถึง** | Dashboard (หน่วยงาน), Assigned Complaints, Reports (หน่วยงาน), Notifications |
| **ข้อมูลที่มองเห็น** | เรื่องทั้งหมดของหน่วยงานตน, สถิติของหน่วยงาน |
| **ข้อจำกัด** | เหมือน Agency Officer — เห็นเฉพาะเรื่องของหน่วยงานตนเอง |

### Role 7: ผู้บริหารจังหวัด (`executive`)

| หัวข้อ | รายละเอียด |
|--------|-----------|
| **คำอธิบาย** | ผู้ว่าราชการจังหวัด รองผู้ว่าฯ นายอำเภอ — ดูเฉพาะ Dashboard และรายงาน |
| **สิทธิ์ที่ทำได้** | ดู Dashboard ภาพรวมทั้งจังหวัด, ดูรายงานสรุป, Export รายงาน |
| **หน้าจอที่เข้าถึง** | Dashboard (Full), Reports (Full) |
| **ข้อมูลที่มองเห็น** | สถิติภาพรวมทั้งหมด, รายการเรื่องร้องเรียน (ไม่เห็นข้อมูลส่วนบุคคลผู้ร้อง) |
| **ข้อจำกัด** | ดูอย่างเดียว (Read-only), ไม่สามารถสร้าง/แก้ไข/ส่งต่อเรื่อง, ไม่เห็นข้อมูลส่วนบุคคลผู้ร้อง |

### Role 8: ประชาชนผู้ร้องเรียน (`public`)

| หัวข้อ | รายละเอียด |
|--------|-----------|
| **คำอธิบาย** | ประชาชนทั่วไปที่ยื่นเรื่อง/ติดตามสถานะ **แบบไม่สมัครสมาชิก (Guest)** — ไม่ต้อง Login |
| **สิทธิ์ที่ทำได้** | ยื่นเรื่องร้องเรียนใหม่ผ่านหน้า Public, ตรวจสอบสถานะเรื่องด้วยเลขที่เรื่อง |
| **หน้าจอที่เข้าถึง** | หน้ายื่นเรื่อง Public (`/public/complaints/new`), หน้าติดตามสถานะ (`/public/track`) |
| **ข้อมูลที่มองเห็น** | สถานะเรื่องร้องเรียนกว้างๆ (เช่น "กำลังดำเนินการ", "ปิดเรื่องแล้ว") — ไม่เห็นรายละเอียดภายใน |
| **ข้อจำกัด** | ไม่ต้อง Login, เห็นเฉพาะสถานะเรื่องของตนเอง, ไม่เห็นข้อมูลภายในหรือเรื่องของคนอื่น |

> 💡 ประชาชนที่ต้องการดูประวัติเรื่องที่เคยยื่นทั้งหมด/ตรวจสถานะโดยไม่ต้องจำเลขที่เรื่อง สามารถ **สมัครสมาชิก (Citizen)** ได้ — ดู §12

## 3. Permission Matrix

ตารางแสดงสิทธิ์ของแต่ละ Role ต่อ Action ต่างๆ ในระบบ:

✅ = ทำได้ &nbsp; ⚠️ = ทำได้บางส่วน &nbsp; ❌ = ทำไม่ได้

| ดูรายละเอียดเรื่อง | ✅ | ✅ | ✅ | ✅ | ⚠️ | ⚠️ | ⚠️ | ⚠️ |
| แก้ไขเรื่อง | ✅ | ✅ | ✅ | ✅ | ❌ | ❌ | ❌ | ❌ |
| จัดประเภท/จัดพื้นที่ | ✅ | ✅ | ✅ | ✅ | ❌ | ❌ | ❌ | ❌ |

> - **Agency Officer/Head**: ดูได้เฉพาะเรื่องที่ส่งมาให้หน่วยงานตนเอง
> - **Executive**: ดูรายการได้ แต่ไม่เห็นข้อมูลส่วนบุคคลผู้ร้อง
> - **Public**: สร้างเรื่องใหม่ได้ผ่านหน้า Public เท่านั้น, ดูสถานะได้เฉพาะด้วยเลขที่เรื่อง

## 5. Data Visibility Rules

ข้อมูลบางประเภทต้องจำกัดการเข้าถึงตาม Role:

### 5.1 ข้อมูลส่วนบุคคลผู้ร้องเรียน

| ข้อมูล | Super Admin | Admin | Officer | Chief | Agency Officer | Agency Head | Executive | Public |
|--------|:-----------:|:-----:|:-------:|:-----:|:--------------:|:-----------:|:---------:|:------:|
| ชื่อ-นามสกุลผู้ร้อง | ✅ | ✅ | ✅ | ✅ | ❌ | ❌ | ❌ | ❌ |
| เลขบัตรประชาชน | ✅ | ✅ | ✅ | ✅ | ❌ | ❌ | ❌ | ❌ |
| ที่อยู่ | ✅ | ✅ | ✅ | ✅ | ❌ | ❌ | ❌ | ❌ |
| เบอร์โทร | ✅ | ✅ | ✅ | ✅ | ❌ | ❌ | ❌ | ❌ |

> **เหตุผล**: ข้อมูลส่วนบุคคลผู้ร้องเรียนเป็นข้อมูลอ่อนไหว (Sensitive Data) ตาม PDPA ต้องจำกัดให้เฉพาะเจ้าหน้าที่ศูนย์ดำรงธรรมที่เป็นผู้รับเรื่อง

#### 5.1.1 เรื่องที่ "ปกปิดตัวตน" (Anonymous) ⭐

**หลักการ** (CF-2, §12.1):

1. ฟอร์มยื่นเรื่อง (ทั้ง Member, Guest และฝั่ง Staff) มี Checkbox **"ปกปิดตัวตน (ไม่เปิดเผยข้อมูลผู้ร้องต่อเจ้าหน้าที่)"**
2. เมื่อเลือก `is_anonymous = true`:
   - ข้อมูลผู้ร้อง (`complainant_name`, `complainant_id_card`, `complainant_phone`, `complainant_address`, `complainant_email`) **ยังถูกบันทึกตามปกติ** ลง `complaints` (⚠️ `complainant_phone` บังคับกรอกเสมอ ตาม BR-16 — แต่ถูก mask เช่นกัน)
   - หากเป็นสมาชิก `citizen_id` **ยังถูกบันทึก** (ผูกภายใน)
3. API ทุกตัวที่ staff เรียก (list, detail, timeline, assignment, export, dashboard) จะ **ไม่ส่ง** ฟิลด์ข้อมูลผู้ร้องและ `citizen_id` ออกไป และส่งธง `is_anonymous = true` แทน เพื่อให้ UI แสดงป้าย "ปกปิดตัวตน"
4. การ mask ทำที่ **ระดับ Backend Query/Serializer** ไม่ใช่ซ่อนที่ Frontend (สอดคล้อง S-3, S-4)

**ใครเห็นอะไร** เมื่อ `is_anonymous = true`:

| ข้อมูล | Super Admin | Admin | Officer | Chief | Agency* | Executive | เจ้าของเรื่อง (citizen) |
|--------|:-----------:|:-----:|:-------:|:-----:|:-------:|:---------:|:----------------------:|
| ชื่อ/เลขบัตร/ที่อยู่/เบอร์ ผู้ร้อง | ⚠️ reveal | ❌ | ❌ | ❌ | ❌ | ✅ (ของตน) |
| `citizen_id` (บัญชีผู้ยื่น) | ⚠️ reveal | ❌ | ❌ | ❌ | ❌ | — |

> - **mask ที่ Backend** (Query/Serializer) ทุก Role รวม officer/chief/admin — ต่างจากเรื่องปกติ
> - ข้อมูลยังคงถูกบันทึกใน DB ครบถ้วน (ไม่ลบ)

**การเปิดเผยตัวตน (Identity Reveal) — เฉพาะ super_admin** (CKD-3):

| ขั้นตอน | รายละเอียด |
|---------|-----------|
| 1 | super_admin กดปุ่ม "เปิดเผยตัวตน" ที่หน้า Complaint Detail (แสดงเฉพาะเรื่อง anonymous) |
| 2 | ระบบบังคับกรอก **เหตุผล (reason)** ก่อนยืนยัน |
| 3 | `POST /api/complaints/:id/reveal-identity` → ตอบกลับข้อมูลผู้ร้องที่ถูก unmask **ครั้งเดียว** |
| 4 | ระบบบันทึกลงตาราง `anonymous_reveal_logs` (ใคร, เรื่องใด, เหตุผล, เมื่อใด) + เขียน `audit_logs` (action `REVEAL_IDENTITY`) |
| 5 | การเปิดเผยไม่เปลี่ยนค่า `is_anonymous` (เรื่องยังคงสถานะปกปิดสำหรับ Role อื่น) |

**ผลกระทบต่อส่วนอื่น:**

| ส่วน | ผลกระทบ |
|------|---------|
| Public/Member Tracking | ไม่เปลี่ยน — ผู้ร้องติดตามเรื่องตนเองได้ตามปกติ |
| Assignment ไปหน่วยงาน | เดิม agency ไม่เห็นข้อมูลส่วนบุคคลอยู่แล้ว → ไม่เปลี่ยน |
| Dashboard/Report/Export | ต้องไม่หลุดข้อมูลผู้ร้องของเรื่อง anonymous (ใช้ field ที่ mask แล้ว) |
| Duplicate check (FR-013) | เจ้าหน้าที่ค้นซ้ำด้วยชื่อ/เบอร์ไม่ได้สำหรับเรื่อง anonymous — ยอมรับข้อจำกัดนี้ |

> Schema `anonymous_reveal_logs`: `05` §4.18 · API: `06` §7.4

### 5.2 ขอบเขตเรื่องร้องเรียนที่มองเห็น

| Role | ขอบเขตเรื่องที่เห็น |
|------|-------------------|
| Super Admin, Admin | เรื่องทั้งหมดในระบบ |
| Officer, Chief | เรื่องทั้งหมดในระบบ |

## 6. Complaint Ownership Rules

กฎเกณฑ์ว่า Role ไหนเป็น "เจ้าของ" เรื่องร้องเรียนในแต่ละขั้นตอน:

| ขั้นตอน | เจ้าของเรื่อง | ผู้มีสิทธิ์ดำเนินการ |
|---------|--------------|---------------------|
| รับเรื่อง → คัดกรอง | ศูนย์ดำรงธรรม | Officer, Chief |
| คัดกรอง → ส่งต่อหน่วยงาน | ศูนย์ดำรงธรรม | Officer, Chief |
| ส่งต่อแล้ว → รับเรื่อง | หน่วยงาน | Agency Officer, Agency Head |
| กำลังดำเนินการ → อัปเดตผล | หน่วยงาน | Agency Officer, Agency Head |
| ส่งผลกลับ → ตรวจสอบ | ศูนย์ดำรงธรรม | Officer, Chief |
| ตรวจสอบ → ปิดเรื่อง | ศูนย์ดำรงธรรม | Officer, Chief |

## 8. Security Considerations

| # | ข้อพิจารณา | รายละเอียด |
|---|-----------|-----------|
| S-1 | **Authentication** | ทุก Role (ยกเว้น Public) ต้อง Login ด้วย JWT Token |
| S-2 | **Authorization** | ทุก API ต้องตรวจสอบ Role ก่อนอนุญาต |
| S-3 | **Data Filtering** | Backend ต้อง Filter ข้อมูลตาม Role ที่ Query Level (ไม่ใช่แค่ซ่อนที่ Frontend) |
| S-4 | **Personal Data** | API ต้องไม่ส่งข้อมูลส่วนบุคคลผู้ร้องให้ Role ที่ไม่มีสิทธิ์ |
| S-5 | **Token Expiry** | JWT Token ควร Expire ภายใน 8 ชั่วโมง (1 วันทำงาน) |
| S-6 | **Password Policy** | รหัสผ่านต้องมีอย่างน้อย 8 ตัวอักษร |
| S-7 | **Brute Force** | จำกัดจำนวนครั้ง Login ผิดติดต่อกัน (เช่น Lock 15 นาทีหลังผิด 5 ครั้ง) |
| S-8 | **Audit** | บันทึก Login/Logout ของทุกผู้ใช้เข้า Audit Log |

## 12. Citizen Membership & Anonymous Complaint (ฟีเจอร์รวม) ⭐

> **เอกสารนี้คือบ้านหลัก (Single Source of Truth) ของ 3 ฟีเจอร์** ที่เพิ่มหลัง Phase 0:
> 1. ระบบสมาชิกประชาชน (Citizen Membership)
> 2. การยื่นเรื่องแบบปกปิดตัวตน (Anonymous Complaint)
> 3. รองรับการยื่นเรื่องทั้งแบบสมาชิกและไม่สมาชิก (Member & Guest Submission)
>
> **citizen ไม่ใช่ Role ใน `users`/`roles`** แต่เป็น **Auth Principal แยก** (ตาราง `citizens`, JWT คนละชุด, middleware `citizenAuth`)

### 12.1 ภาพรวมฟีเจอร์ (Feature Overview)

| # | ฟีเจอร์ | รายละเอียด |
|---|---------|-----------|
| CF-1 | **สมัครสมาชิกประชาชน** | ประชาชนสร้างบัญชีเอง (อีเมล + รหัสผ่าน) เพื่อ Login, ดูเรื่องที่เคยยื่น, ตรวจสอบสถานะ, แก้ไขข้อมูลส่วนตัว |
| CF-2 | **ปกปิดตัวตน (Anonymous)** | ในขั้นตอนยื่นเรื่อง เลือก "ปกปิดตัวตน" ได้ — ข้อมูลส่วนตัว **ยังถูกบันทึกในฐานข้อมูล** แต่ **ไม่แสดง/ไม่ถูกแนบไปให้เจ้าหน้าที่ทุก Role** ใน UI |
| CF-3 | **ยื่นได้ทั้งสมาชิกและไม่สมาชิก** | เรื่องใหม่ยื่นได้ทั้ง (ก) แบบสมาชิก (Login — ผูกกับบัญชี ติดตามได้ในหน้า "เรื่องของฉัน") และ (ข) แบบไม่สมาชิก (Guest — ติดตามด้วยเลขที่เรื่องเท่านั้น) |

```

                                     │
                ┌────────────────────┴────────────────────┐
                ▼                                          ▼
        แบบสมาชิก (Member)                         แบบไม่สมาชิก (Guest)
   - ต้อง Login ด้วยบัญชี citizen              - ไม่ต้อง Login
   - เรื่องผูกกับ citizen_id                   - citizen_id = NULL
   - ติดตามได้ใน "เรื่องของฉัน"                - ติดตามด้วยเลขที่เรื่อง
                │                                          │
                └────────────────────┬────────────────────┘
                                     ▼

                                     │

                   ▼                                     ▼
        is_anonymous = false                  is_anonymous = true
   - เจ้าหน้าที่เห็นข้อมูลผู้ร้อง          - ข้อมูลผู้ร้องถูก "mask" จาก
     ตามกฎ Role (§5.1)                       เจ้าหน้าที่ทุก Role (§5.1.1)
                                            - ข้อมูลยังเก็บใน DB
                                            - citizen_id ยังผูกภายใน
                                            - super_admin "เปิดเผยตัวตน" ได้
```

### 12.2 Key Decisions (อนุมัติแล้ว)

| # | การตัดสินใจ | เหตุผล / ที่มา |
|---|-----------|----------------|
| CKD-1 | **แยกตาราง `citizens` ออกจาก `users` + ระบบ Auth แยก** (คนละ JWT, คนละ endpoint namespace) | RBAC ของ staff ไม่ปนกับบัญชีประชาชน ลดความเสี่ยงด้านสิทธิ์ และขยาย flow ของประชาชนได้อิสระ — ยืนยันโดยผู้ใช้ |
| CKD-2 | **เรื่อง Anonymous ถูก mask จาก staff ทุก Role ใน UI** | ตรงตาม requirement "ไม่แสดงให้เจ้าหน้าที่เห็น" — รวมถึง officer/chief/admin ที่ปกติเห็นข้อมูลส่วนบุคคล |
| CKD-3 | **เฉพาะ `super_admin` เท่านั้นที่ "เปิดเผยตัวตน" เรื่อง Anonymous ได้ + ต้องบันทึก audit ทุกครั้ง** | ใช้เฉพาะกรณีจำเป็นทางกฎหมาย/การตรวจสอบ — ยืนยันโดยผู้ใช้ |
| CKD-4 | **สมาชิกที่ยื่นแบบ Anonymous → ระบบผูก `citizen_id` ภายใน แต่ซ่อนตัวตนจาก staff** | ให้สมาชิกยังติดตามเรื่องใน "เรื่องของฉัน" ได้ ขณะที่ staff ยังมองไม่เห็นตัวตน — ยืนยันโดยผู้ใช้ |
| CKD-5 | **ข้อมูลส่วนตัวของผู้ร้องเก็บที่ตาราง `complaints` (snapshot) เสมอ** ไม่ดึงสดจาก `citizens` ตอนแสดงผล | เรื่องร้องเรียนเป็นหลักฐาน ต้องคงข้อมูล ณ เวลายื่น แม้สมาชิกแก้โปรไฟล์ภายหลัง |
| CKD-6 | **ช่องทาง (channel) ของการยื่นออนไลน์ = "เว็บไซต์"; `source` = `PUBLIC`** ทั้งสมาชิกและ Guest | แยกสมาชิก/Guest ด้วย `citizen_id` (NOT NULL = สมาชิก) ไม่ต้องเพิ่มค่า ENUM ใหม่ |

### 12.3 Citizen Principal — ความสามารถและขอบเขต

| หัวข้อ | รายละเอียด |
|--------|-----------|
| **คำอธิบาย** | ประชาชนที่สมัครสมาชิกเพื่อ Login, ดูเรื่องที่เคยยื่น, ตรวจสถานะ, แก้ไขข้อมูลส่วนตัว |
| **สิทธิ์ที่ทำได้** | สมัคร/เข้าสู่ระบบ, ยื่นเรื่อง (ผูกบัญชี), ดู "เรื่องของฉัน" + สถานะย่อ, แก้ไขข้อมูลส่วนตัว, เปลี่ยนรหัสผ่าน, เลือกปกปิดตัวตนตอนยื่น |
| **หน้าจอที่เข้าถึง** | `/citizen/*` (register, login, my-complaints, profile) — ดู `07` §5.1A |
| **ข้อมูลที่มองเห็น** | เฉพาะเรื่อง **ของตนเอง** (สถานะแบบย่อเหมือน Public Tracking) + ข้อมูลบัญชีตนเอง |
| **ข้อจำกัด** | ไม่เห็นข้อมูลภายในระบบ/เรื่องของคนอื่น, Token เรียกได้เฉพาะ `/api/citizen/*` (เรียก endpoint staff = 403) |

| # | ความสามารถ (Capability) | Endpoint หลัก |
|---|------------------------|----------------|
| CC-1 | สมัครสมาชิก | `POST /api/citizen/auth/register` |
| CC-2 | เข้าสู่ระบบ / ออกจากระบบ | `POST /api/citizen/auth/login`, `/logout` |
| CC-3 | ดูข้อมูลบัญชีตนเอง | `GET /api/citizen/auth/me` |
| CC-4 | แก้ไขข้อมูลส่วนตัว | `PUT /api/citizen/profile` |
| CC-5 | เปลี่ยนรหัสผ่าน | `PUT /api/citizen/auth/change-password` |
| CC-6 | ยื่นเรื่องร้องเรียน (ผูกบัญชี) | `POST /api/citizen/complaints` |
| CC-7 | ดูรายการเรื่องที่เคยยื่น ("เรื่องของฉัน") | `GET /api/citizen/complaints` |
| CC-8 | ดูสถานะ/Timeline ของเรื่องตนเอง | `GET /api/citizen/complaints/:complaint_number` |

### 12.4 Roles & Auth Summary

- **`citizen` ไม่ใช่ Role ใน `users`/`roles`** แต่เป็น **Auth Principal แยก** (ตาราง `citizens`, JWT คนละชุด, middleware `citizenAuth`)
- Token ของ citizen ใช้ namespace `/api/citizen/*` เท่านั้น — เรียก endpoint ของ staff ไม่ได้ (และกลับกัน)
- Role `public` เดิม (§1, §2) ยังคงไว้สำหรับ "การยื่น/ติดตามแบบ Guest (ไม่ Login)"

### 12.5 Security เพิ่มเติมสำหรับ Citizen

| # | ข้อพิจารณา | รายละเอียด |
|---|-----------|-----------|
| S-9 | **Separate Auth Scope** | citizen JWT และ staff JWT แยก namespace โดยสมบูรณ์ (ตรวจที่ middleware) |
| S-10 | **Password Policy** | รหัสผ่านสมาชิก ≥ 8 ตัวอักษร, เข้ารหัส bcrypt |
| S-11 | **Self-data only** | ทุก endpoint ของ citizen ต้อง filter ด้วย `citizen_id` ของ token ที่ระดับ Query |

### 12.6 Out of Scope (ฟีเจอร์นี้ — Phase แรก)

| # | รายการ | หมายเหตุ |
|---|--------|---------|
| CO-1 | Email verification / OTP ตอนสมัคร | เพิ่มภายหลัง (ต้องมี Email/SMS gateway) |
| CO-2 | แจ้งเตือนสมาชิกผ่าน Email/SMS/LINE เมื่อสถานะเปลี่ยน | คง C-5 (in-app เท่านั้น) — สมาชิกเข้ามาเช็คเอง |
| CO-3 | แจ้งเตือน in-app ถึงสมาชิก (`notifications` ผูก citizen) | พิจารณา Phase ถัดไป — ปัจจุบัน `notifications.user_id` ผูกกับ staff เท่านั้น |
| CO-4 | Social login (Google/ThaID) | อนาคต |
| CO-5 | ลบบัญชีสมาชิก (Right to be forgotten ตาม PDPA) | กำหนดนโยบายภายหลัง — Phase แรกใช้ Soft Delete (`is_active`) |
