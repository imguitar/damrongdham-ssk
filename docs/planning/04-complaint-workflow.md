# 04 — Complaint Workflow: ระบบร้องเรียนศูนย์ดำรงธรรมจังหวัด

> เอกสารฉบับนี้เป็นส่วนหนึ่งของขั้นตอน Planning Only — ยังไม่มีการเขียน Code
> อ้างอิงจาก: `01-system-overview.md`, `02-requirements.md`, `03-roles-permissions.md`

## 1. Workflow Overview

เรื่องร้องเรียนจะเดินทางผ่าน **11 สถานะหลัก** โดยมี 2 ฝ่ายหลักที่ทำงานร่วมกัน:

- **ศูนย์ดำรงธรรม** (Officer, Chief) — รับเรื่อง คัดกรอง ส่งต่อ ตรวจผล ปิดเรื่อง
- **หน่วยงานที่รับผิดชอบ** (Agency Officer, Agency Head) — รับเรื่อง ดำเนินการ ส่งผลกลับ

```
ประชาชน                    ศูนย์ดำรงธรรม                    หน่วยงาน
   │                            │                              │
   │── ยื่นเรื่อง ──────────►  รับเรื่อง (NEW)                 │
   │                            │                              │
   │                       คัดกรอง (SCREENING)                 │
   │                            │                              │
   │                      ┌─────┴─────┐                        │
   │                   ผ่าน        ไม่ผ่าน                      │
   │                      │       (REJECTED)                   │
   │                      │                                    │
   │                 ส่งต่อหน่วยงาน (ASSIGNED)                  │
   │                            │──────────────────────────►   │
   │                            │                         รับเรื่อง (ACCEPTED)
   │                            │                              │
   │                            │                        ดำเนินการ (IN_PROGRESS)
   │                            │                              │
   │                            │                         ส่งผลกลับ (RESOLVED)
   │                            │  ◄──────────────────────────│
   │                       ตรวจผล (REVIEWING)                  │
   │                            │                              │
   │                      ┌─────┴─────┐                        │
   │                   ผ่าน        ส่งกลับแก้ไข                 │
   │                      │       (IN_PROGRESS)                │
   │                      │                                    │
   │                  ปิดเรื่อง (CLOSED)                        │
   │  ◄── แจ้งผล ────────│                                    │
   │                            │                              │
```

## 2. Complaint Status List

ระบบมี **11 สถานะ** สำหรับเรื่องร้องเรียน:

| # | Status Code | ชื่อสถานะ (ไทย) | ชื่อสถานะ (อังกฤษ) | สี | ประเภท |
|---|------------|-----------------|-------------------|-----|--------|
| 1 | `NEW` | รับเรื่องใหม่ | New | 🔵 น้ำเงิน | ปกติ |
| 2 | `SCREENING` | กำลังคัดกรอง | Screening | 🟡 เหลือง | ปกติ |
| 3 | `ASSIGNED` | ส่งต่อหน่วยงานแล้ว | Assigned | 🟠 ส้ม | ปกติ |
| 4 | `ACCEPTED` | หน่วยงานรับเรื่องแล้ว | Accepted | 🟣 ม่วง | ปกติ |
| 5 | `IN_PROGRESS` | กำลังดำเนินการ | In Progress | 🔵 ฟ้า | ปกติ |
| 6 | `RESOLVED` | ส่งผลดำเนินการแล้ว | Resolved | 🟢 เขียวอ่อน | ปกติ |
| 7 | `REVIEWING` | กำลังตรวจสอบผล | Reviewing | 🟡 เหลือง | ปกติ |
| 8 | `CLOSED` | ปิดเรื่อง | Closed | 🟢 เขียว | สิ้นสุด |
| 9 | `REJECTED` | ปฏิเสธเรื่อง | Rejected | 🔴 แดง | สิ้นสุด |
| 10 | `RETURNED` | ส่งคืนเรื่อง | Returned | ⚪ เทา | ผิดปกติ |
| 11 | `OVERDUE` | เกินกำหนด | Overdue | 🔴 แดง | ระบบกำหนด |

> **หมายเหตุ**: `OVERDUE` ไม่ใช่สถานะจริง แต่เป็น **Flag** ที่ระบบคำนวณอัตโนมัติจาก Due Date เรื่องที่เกินกำหนดจะมี Flag `is_overdue = true` แต่ยังคงสถานะเดิม (เช่น `IN_PROGRESS` + Overdue)

## 3. Status Description

### 3.1 NEW — รับเรื่องใหม่

| หัวข้อ | รายละเอียด |
|--------|-----------|
| **คำอธิบาย** | เรื่องร้องเรียนถูกบันทึกเข้าสู่ระบบแล้ว รอเจ้าหน้าที่คัดกรอง |
| **สร้างโดย** | Officer/Chief (บันทึกเอง, source=STAFF) · สมาชิกประชาชน (Login, source=PUBLIC, ผูก `citizen_id`) · Guest (ไม่ Login, source=PUBLIC) |
| **ข้อมูลที่ต้องมี (บังคับ)** | ชื่อเรื่องร้องเรียน, รายละเอียด, ประเภทงานบริการ, ลักษณะเรื่อง, ประเภทผู้ร้องเรียน, ช่องทางรับเรื่อง, **เบอร์โทรผู้ร้อง** (BR-16) |
| **ข้อมูลที่บันทึกได้ (Optional)** | ชื่อ/เลขบัตร/ที่อยู่/อีเมลผู้ร้อง, จุดเกิดเหตุ (จังหวัด/อำเภอ/ตำบล/รหัสไปรษณีย์/สถานที่), พิกัดแผนที่ (GPS/Leaflet), ไฟล์แนบหลักฐาน |
| **ปกปิดตัวตน** | ทุกช่องทางเลือก `is_anonymous` ได้ — ถ้าเลือก ข้อมูลผู้ร้อง + `citizen_id` จะถูก mask จากเจ้าหน้าที่ทุก Role (ดู `03` §5.1.1) |
| **Action ที่ทำได้** | เริ่มคัดกรอง, แก้ไขข้อมูล |
| **ระยะเวลาสูงสุด** | ต้องเริ่มคัดกรองภายใน 1 วันทำการ |

### 3.2 SCREENING — กำลังคัดกรอง

| หัวข้อ | รายละเอียด |
|--------|-----------|
| **คำอธิบาย** | เจ้าหน้าที่กำลังตรวจสอบข้อมูล จัดประเภท กำหนดพื้นที่ ตรวจเรื่องซ้ำ |
| **ผู้รับผิดชอบ** | Officer, Chief |
| **Action ที่ทำได้** | จัดประเภท, กำหนดพื้นที่, กำหนด Priority, ส่งต่อหน่วยงาน, ปฏิเสธเรื่อง |
| **ระยะเวลาสูงสุด** | ต้องดำเนินการภายใน 1 วันทำการ |

### 3.3 ASSIGNED — ส่งต่อหน่วยงานแล้ว

| หัวข้อ | รายละเอียด |
|--------|-----------|
| **คำอธิบาย** | เรื่องถูกส่งต่อไปยังหน่วยงานที่รับผิดชอบแล้ว รอหน่วยงานรับเรื่อง |
| **ผู้รับผิดชอบ** | Officer, Chief (ผู้ส่งต่อ) |
| **ข้อมูลที่บันทึก** | หน่วยงานปลายทาง, วันที่ส่งต่อ, Due Date, หมายเหตุ |
| **Action ที่ทำได้** | (รอหน่วยงานรับเรื่อง) |
| **Notification** | แจ้งเตือนหน่วยงานปลายทางทันที |

### 3.4 ACCEPTED — หน่วยงานรับเรื่องแล้ว

| หัวข้อ | รายละเอียด |
|--------|-----------|
| **คำอธิบาย** | หน่วยงานกดยืนยันว่าได้รับเรื่องแล้ว พร้อมเริ่มดำเนินการ |
| **ผู้รับผิดชอบ** | Agency Officer, Agency Head |
| **Action ที่ทำได้** | เริ่มดำเนินการ, ส่งคืนเรื่อง (กรณีส่งผิด) |

### 3.5 IN_PROGRESS — กำลังดำเนินการ

| หัวข้อ | รายละเอียด |
|--------|-----------|
| **คำอธิบาย** | หน่วยงานกำลังดำเนินการแก้ไขปัญหาตามเรื่องร้องเรียน |
| **ผู้รับผิดชอบ** | Agency Officer, Agency Head |
| **Action ที่ทำได้** | อัปเดตความคืบหน้า, แนบไฟล์, ส่งผลดำเนินการ |
| **ระยะเวลา** | ตาม SLA ของประเภทเรื่อง |

### 3.6 RESOLVED — ส่งผลดำเนินการแล้ว

| หัวข้อ | รายละเอียด |
|--------|-----------|
| **คำอธิบาย** | หน่วยงานส่งผลดำเนินการกลับมาให้ศูนย์ดำรงธรรมตรวจสอบ |
| **ผู้รับผิดชอบ** | Agency Officer, Agency Head (ผู้ส่งผล) |
| **ข้อมูลที่บันทึก** | สรุปผลดำเนินการ, ไฟล์หลักฐาน |
| **Notification** | แจ้งเตือน Officer/Chief ที่ศูนย์ดำรงธรรม |

### 3.7 REVIEWING — กำลังตรวจสอบผล

| หัวข้อ | รายละเอียด |
|--------|-----------|
| **คำอธิบาย** | ศูนย์ดำรงธรรมกำลังตรวจสอบผลดำเนินการจากหน่วยงาน |
| **ผู้รับผิดชอบ** | Officer, Chief |
| **Action ที่ทำได้** | อนุมัติผล → ปิดเรื่อง, ส่งกลับแก้ไข → กลับไป IN_PROGRESS |

### 3.8 CLOSED — ปิดเรื่อง

| หัวข้อ | รายละเอียด |
|--------|-----------|
| **คำอธิบาย** | เรื่องร้องเรียนดำเนินการเสร็จสิ้น ปิดเรื่องแล้ว |
| **ผู้รับผิดชอบ** | Officer, Chief (ผู้ปิดเรื่อง) |
| **ข้อมูลที่บันทึก** | สรุปผลการปิดเรื่อง, วันที่ปิด |
| **สถานะสุดท้าย** | ไม่สามารถเปลี่ยนสถานะต่อได้ (Final) |

### 3.9 REJECTED — ปฏิเสธเรื่อง

| หัวข้อ | รายละเอียด |
|--------|-----------|
| **คำอธิบาย** | เรื่องถูกปฏิเสธเนื่องจากไม่อยู่ในอำนาจหน้าที่ ข้อมูลไม่เพียงพอ หรือเป็นเรื่องซ้ำ |
| **ผู้รับผิดชอบ** | Officer, Chief |
| **ข้อมูลที่ต้องบันทึก** | เหตุผลในการปฏิเสธ (บังคับ) |
| **สถานะสุดท้าย** | ไม่สามารถเปลี่ยนสถานะต่อได้ (Final) |

### 3.10 RETURNED — ส่งคืนเรื่อง

| หัวข้อ | รายละเอียด |
|--------|-----------|
| **คำอธิบาย** | หน่วยงานส่งคืนเรื่องกลับมาที่ศูนย์ดำรงธรรม (กรณีส่งผิดหน่วยงาน หรือไม่อยู่ในอำนาจ) |
| **ผู้รับผิดชอบ** | Agency Officer, Agency Head |
| **ข้อมูลที่ต้องบันทึก** | เหตุผลในการส่งคืน (บังคับ) |
| **Action ถัดไป** | ศูนย์ดำรงธรรมต้องส่งต่อหน่วยงานใหม่ → กลับไปสถานะ SCREENING |

|---|---------|---------|---------|---------|-------------|
| T-01 | `NEW` | `SCREENING` | Officer, Chief | เจ้าหน้าที่เริ่มตรวจสอบเรื่อง | — |
| T-02 | `SCREENING` | `ASSIGNED` | Officer, Chief | จัดประเภทแล้ว + เลือกหน่วยงานแล้ว | แจ้งหน่วยงาน |
| T-03 | `SCREENING` | `REJECTED` | Officer, Chief | ระบุเหตุผลปฏิเสธ (บังคับ) | — |

| T-06 | `ACCEPTED` | `IN_PROGRESS` | Agency Officer/Head | เริ่มดำเนินการ | — |
| T-07 | `ACCEPTED` | `RETURNED` | Agency Officer/Head | ส่งคืนพร้อมเหตุผล | แจ้ง Officer |

| T-09 | `RESOLVED` | `REVIEWING` | Officer, Chief | เจ้าหน้าที่เริ่มตรวจสอบผล | — |
| T-10 | `REVIEWING` | `CLOSED` | Officer, Chief | อนุมัติผล + สรุปผลปิดเรื่อง | แจ้งหน่วยงาน |
| T-11 | `REVIEWING` | `IN_PROGRESS` | Officer, Chief | ส่งกลับแก้ไขพร้อมหมายเหตุ | แจ้งหน่วยงาน |
| T-12 | `RETURNED` | `SCREENING` | Officer, Chief | รับเรื่องคืนแล้ว เริ่มคัดกรองใหม่ | — |

### Transition ที่ห้าม (ไม่อนุญาต)

| ห้ามเปลี่ยนจาก | ไปยัง | เหตุผล |
|---------------|------|--------|
| `CLOSED` | สถานะใดก็ตาม | เรื่องที่ปิดแล้วไม่สามารถเปิดใหม่ได้ |
| `REJECTED` | สถานะใดก็ตาม | เรื่องที่ปฏิเสธแล้วไม่สามารถดำเนินการต่อ |
| `IN_PROGRESS` | `ASSIGNED` | ไม่สามารถย้อนกลับไปสถานะก่อนรับเรื่อง |
| `NEW` | `CLOSED` | ไม่สามารถปิดเรื่องโดยไม่ผ่านกระบวนการ |

## 5. Main Workflow Steps

### Step 0: ช่องทางการยื่นเรื่อง (Member / Guest / Staff)

เรื่องใหม่ยื่นได้ 3 รูปแบบ โดยใช้ Workflow เดียวกัน (เริ่มที่ `NEW`) ต่างกันแค่ `source`/`citizen_id`:

| มิติ | สมาชิก (Member) | ไม่สมาชิก (Guest) | เจ้าหน้าที่บันทึก (Staff) |
|------|-----------------|--------------------|---------------------------|
| ต้อง Login | ✅ (citizen JWT) | ❌ | ✅ (staff JWT) |
| Endpoint | `POST /api/citizen/complaints` | `POST /api/public/complaints` | `POST /api/complaints` |
| `source` | `PUBLIC` | `PUBLIC` | `STAFF` |
| `citizen_id` | NOT NULL | NULL | NULL |
| ปกปิดตัวตนได้ | ✅ | ✅ | ✅ |
| ติดตามเรื่อง | "เรื่องของฉัน" + เลขที่เรื่อง | เลขที่เรื่องเท่านั้น | ในระบบ staff |
| Pre-fill ข้อมูลผู้ร้อง | จากโปรไฟล์ citizen (แก้ได้ก่อนส่ง) | กรอกเอง | กรอกเอง |

- ทุกช่องทางสร้างเรื่องด้วยสถานะเริ่มต้น `NEW`
- เรื่องที่ผูก `citizen_id` หากเลือก anonymous: ข้อมูล + การผูกบัญชียัง mask จาก staff (CKD-4)
- รายละเอียดสิทธิ์/Auth ของ Citizen และการปกปิดตัวตน: ดู `03` §12 และ §5.1.1

### Step 1: การรับเรื่องร้องเรียน

| หัวข้อ | รายละเอียด |
|--------|-----------|
| **ผู้กระทำ** | Officer/Chief (บันทึกเอง) · สมาชิกประชาชน (Login) · Guest (ไม่ Login) |
| **สถานะ** | → `NEW` |
| **ข้อมูลที่ต้องกรอก (บังคับ)** | ชื่อเรื่อง, รายละเอียด, ประเภทงานบริการ, ลักษณะเรื่อง, ประเภทผู้ร้องเรียน, ช่องทาง, **เบอร์โทรผู้ร้อง** (BR-16) · สมาชิก pre-fill ข้อมูลผู้ร้องได้ · เลือกปกปิดตัวตน (optional) |
| **ข้อมูลจุดเกิดเหตุ (Optional)** | จังหวัด (default ศรีสะเกษ), อำเภอ, ตำบล, รหัสไปรษณีย์, สถานที่เกิดเหตุ, พิกัดแผนที่ (GPS หรือเลือกบน Leaflet), ไฟล์แนบหลักฐาน |
| **ข้อมูลที่ระบบสร้างอัตโนมัติ** | เลขที่เรื่อง (DC-YYYYMM-XXXX), วันที่รับเรื่อง, ผู้รับเรื่อง |
| **ช่องทางรับเรื่อง** | จดหมาย, ลงพื้นที่/หน่วยเคลื่อนที่เร็ว, ตู้ราชสีห์, สายด่วน 1567, Walk In, เว็บไซต์, แอปฯ MOI1567, ผ่านหน่วยงานอื่น, อื่น ๆ |

### Step 2: การตรวจสอบข้อมูลเบื้องต้น (คัดกรอง)

| หัวข้อ | รายละเอียด |
|--------|-----------|
| **ผู้กระทำ** | Officer, Chief |
| **สถานะ** | `NEW` → `SCREENING` |
| **สิ่งที่ต้องตรวจสอบ** | ข้อมูลครบถ้วนหรือไม่, เป็นเรื่องซ้ำหรือไม่, อยู่ในอำนาจหน้าที่หรือไม่ |
| **ผลลัพธ์ที่เป็นไปได้** | ผ่าน → ไปขั้นตอน 3 / ไม่ผ่าน → `REJECTED` |

### Step 3: การจัดประเภทเรื่องร้องเรียน

| หัวข้อ | รายละเอียด |
|--------|-----------|
| **ผู้กระทำ** | Officer, Chief |
| **สถานะ** | ยังอยู่ `SCREENING` |
| **Action** | เลือกประเภทเรื่อง (Category), กำหนด Priority (สูง/ปานกลาง/ต่ำ) |
| **ผลกระทบ** | ระบบคำนวณ Due Date อัตโนมัติจาก SLA ของ Category |

### Step 4: การกำหนดพื้นที่

| หัวข้อ | รายละเอียด |
|--------|-----------|
| **ผู้กระทำ** | Officer, Chief |
| **สถานะ** | ยังอยู่ `SCREENING` |
| **Action** | เลือกอำเภอ/ตำบลที่เกี่ยวข้อง |

### Step 5: การส่งต่อหน่วยงานรับผิดชอบ

| หัวข้อ | รายละเอียด |
|--------|-----------|
| **ผู้กระทำ** | Officer, Chief |
| **สถานะ** | `SCREENING` → `ASSIGNED` |
| **Action** | เลือกหน่วยงานที่รับผิดชอบ, ระบุหมายเหตุ (ถ้ามี) |
| **ข้อมูลที่บันทึก** | หน่วยงาน, วันที่ส่งต่อ, Due Date (คำนวณจาก SLA) |
| **Notification** | ✅ แจ้งเตือนเจ้าหน้าที่หน่วยงานปลายทาง |

### Step 6: การรับเรื่องโดยหน่วยงาน

| หัวข้อ | รายละเอียด |
|--------|-----------|
| **ผู้กระทำ** | Agency Officer, Agency Head |
| **สถานะ** | `ASSIGNED` → `ACCEPTED` |
| **Action** | กดปุ่ม "รับเรื่อง" เพื่อยืนยัน |
| **ทางเลือก** | กด "ส่งคืน" → `RETURNED` (กรณีส่งผิดหน่วยงาน) |

### Step 7: การดำเนินการและอัปเดตความคืบหน้า

| หัวข้อ | รายละเอียด |
|--------|-----------|
| **ผู้กระทำ** | Agency Officer, Agency Head |
| **สถานะ** | `ACCEPTED` → `IN_PROGRESS` |
| **Action** | อัปเดต Progress Note, แนบไฟล์หลักฐาน (สามารถอัปเดตได้หลายครั้ง) |
| **ระยะเวลา** | ต้องดำเนินการให้เสร็จภายใน Due Date |

### Step 8: การส่งผลดำเนินการ

| หัวข้อ | รายละเอียด |
|--------|-----------|
| **ผู้กระทำ** | Agency Officer, Agency Head |
| **สถานะ** | `IN_PROGRESS` → `RESOLVED` |
| **Action** | กดปุ่ม "ส่งผลดำเนินการ" พร้อมสรุปผลและหลักฐาน |
| **Notification** | ✅ แจ้งเตือน Officer/Chief ที่ศูนย์ดำรงธรรม |

### Step 9: การตรวจสอบผลดำเนินการ

| หัวข้อ | รายละเอียด |
|--------|-----------|
| **ผู้กระทำ** | Officer, Chief |
| **สถานะ** | `RESOLVED` → `REVIEWING` |
| **Action** | ตรวจสอบความสมบูรณ์ของผลดำเนินการ |
| **ผลลัพธ์** | อนุมัติ → `CLOSED` / ส่งกลับแก้ไข → `IN_PROGRESS` |

### Step 10: การปิดเรื่อง

| หัวข้อ | รายละเอียด |
|--------|-----------|
| **ผู้กระทำ** | Officer, Chief |
| **สถานะ** | `REVIEWING` → `CLOSED` |
| **Action** | กดปุ่ม "ปิดเรื่อง" พร้อมสรุปผลการปิด |
| **ข้อมูลที่บันทึก** | สรุปผล, วันที่ปิดเรื่อง, ผู้ปิดเรื่อง |
| **Notification** | ✅ แจ้งเตือนหน่วยงาน |

### Step 11: การแจ้งผลให้ประชาชน

| หัวข้อ | รายละเอียด |

## 7. SLA Rules

| # | กฎ | รายละเอียด |
|---|---|-----------|
| SLA-01 | **SLA เริ่มนับ** | จากวันที่ส่งต่อหน่วยงาน (`ASSIGNED`) ไม่ใช่วันที่รับเรื่อง |
| SLA-02 | **กำหนดตาม Category** | แต่ละประเภทเรื่องมี SLA ต่างกัน (เช่น ที่ดิน = 15 วัน, สิ่งแวดล้อม = 7 วัน) |
| SLA-03 | **Due Date** | คำนวณอัตโนมัติ: `Due Date = วันที่ส่งต่อ + SLA ของ Category` |
| SLA-04 | **ใกล้ครบกำหนด** | ระบบแจ้งเตือนเมื่อเหลือเวลา ≤ 3 วันทำการ (Flag: `is_near_due`) |
| SLA-05 | **เกินกำหนด** | ระบบ Flag อัตโนมัติเมื่อ `วันปัจจุบัน > Due Date` และเรื่องยังไม่ปิด (Flag: `is_overdue`) |
| SLA-06 | **นับเฉพาะวันทำการ** | (Assumption) SLA นับรวมวันหยุดเสาร์-อาทิตย์ด้วย (เพื่อความง่ายใน Phase แรก) |

## 8. Exception Workflow (กรณีผิดปกติ)

### 8.1 กรณีส่งผิดหน่วยงาน

```
ASSIGNED → RETURNED (โดยหน่วยงาน ระบุเหตุผล)
         → SCREENING (โดยศูนย์ รับคืนแล้วคัดกรองใหม่)
         → ASSIGNED (ส่งต่อหน่วยงานใหม่)
```

| ขั้นตอน | ผู้กระทำ | Action |
|---------|---------|--------|
| 1 | Agency Officer/Head | กดปุ่ม "ส่งคืน" พร้อมระบุเหตุผล (บังคับ) |
| 2 | ระบบ | แจ้งเตือน Officer ที่ศูนย์ดำรงธรรม |
| 3 | Officer/Chief | รับเรื่องคืน ตรวจสอบ เลือกหน่วยงานใหม่ ส่งต่ออีกครั้ง |

### 8.2 กรณีเรื่องซ้ำ (Duplicate)

| ขั้นตอน | ผู้กระทำ | Action |
|---------|---------|--------|
| 1 | Officer/Chief | ค้นพบเรื่องซ้ำขณะคัดกรอง (ค้นจากชื่อผู้ร้อง, เบอร์โทร, เนื้อหา) |
| 2 | Officer/Chief | กด "ปฏิเสธ" (REJECTED) พร้อมระบุเหตุผล "เรื่องซ้ำ" และอ้างอิงเลขที่เรื่องเดิม |

### 8.3 กรณีข้อมูลไม่ครบ

| ขั้นตอน | ผู้กระทำ | Action |
|---------|---------|--------|
| 1 | Officer/Chief | ตรวจพบว่าข้อมูลไม่เพียงพอ |
| 2 | Officer/Chief | ทางเลือก A: ติดต่อผู้ร้องเพื่อขอข้อมูลเพิ่ม (นอกระบบ) แล้วเพิ่มเข้าระบบ |
| 3 | Officer/Chief | ทางเลือก B: ปฏิเสธเรื่อง (REJECTED) ระบุเหตุผล "ข้อมูลไม่เพียงพอ" |

### 8.4 กรณีเรื่องเกินกำหนด (Overdue)

```
ระบบตรวจจับอัตโนมัติ ทุกวัน
    → ถ้า วันปัจจุบัน > Due Date และ สถานะไม่ใช่ CLOSED/REJECTED
    → Flag: is_overdue = true
    → แจ้งเตือน Officer, Chief, Agency Officer, Agency Head
```

| ขั้นตอน | ผู้กระทำ | Action |
|---------|---------|--------|
| 1 | ระบบ | ตรวจจับเรื่องเกินกำหนด (ทุกวัน หรือ Real-time) |
| 2 | ระบบ | แจ้งเตือนเจ้าหน้าที่ศูนย์ + หน่วยงาน |
| 3 | Officer/Chief | ติดตามเร่งรัดหน่วยงาน (โทรหรือส่งหนังสือเร่งรัด) |
| 4 | Agency Officer/Head | เร่งดำเนินการและอัปเดตผล |

### 8.5 กรณีเรื่องไม่อยู่ในอำนาจหน้าที่

| ขั้นตอน | ผู้กระทำ | Action |
|---------|---------|--------|
| 1 | Officer/Chief | ตรวจพบว่าเรื่องไม่อยู่ในอำนาจของศูนย์ดำรงธรรมจังหวัด |
| 2 | Officer/Chief | ปฏิเสธเรื่อง (REJECTED) ระบุเหตุผล "ไม่อยู่ในอำนาจหน้าที่" |
| 3 | Officer/Chief | (Optional) แนะนำช่องทางอื่นให้ผู้ร้องในหมายเหตุ |

### 8.6 กรณีผลดำเนินการไม่สมบูรณ์

```
RESOLVED → REVIEWING (ศูนย์เริ่มตรวจ)
         → IN_PROGRESS (ส่งกลับแก้ไข พร้อมหมายเหตุ)
         → RESOLVED (หน่วยงานส่งผลใหม่)
         → REVIEWING → CLOSED
```

| ขั้นตอน | ผู้กระทำ | Action |
|---------|---------|--------|
| 1 | Officer/Chief | ตรวจผลแล้วพบว่าไม่สมบูรณ์ |
| 2 | Officer/Chief | กด "ส่งกลับแก้ไข" พร้อมระบุสิ่งที่ต้องแก้ |
| 3 | ระบบ | สถานะกลับเป็น IN_PROGRESS + แจ้งเตือนหน่วยงาน |
| 4 | Agency Officer/Head | แก้ไขและส่งผลใหม่ |

## 9. Notification Triggers

| # | เหตุการณ์ | ผู้รับการแจ้งเตือน | ข้อความ |
|---|---------|-----------------|--------|
| N-01 | เรื่องใหม่จากประชาชน (Public) | Officer ทุกคน | "มีเรื่องร้องเรียนใหม่จากประชาชน: {เลขที่เรื่อง}" |

| N-04 | หน่วยงานส่งคืนเรื่อง | Officer ผู้ส่งต่อ | "หน่วยงาน {ชื่อ} ส่งคืนเรื่อง {เลขที่} เหตุผล: {เหตุผล}" |
| N-05 | หน่วยงานส่งผลดำเนินการ | Officer, Chief | "หน่วยงาน {ชื่อ} ส่งผลดำเนินการเรื่อง {เลขที่}" |
| N-06 | ศูนย์ส่งกลับแก้ไข | Agency Officer/Head | "เรื่อง {เลขที่} ถูกส่งกลับแก้ไข หมายเหตุ: {หมายเหตุ}" |
| N-07 | เรื่องปิดแล้ว | Agency Officer/Head | "เรื่อง {เลขที่} ปิดเรื่องแล้ว" |
| N-08 | ใกล้ครบกำหนด (≤3 วัน) | Officer, Agency Officer/Head | "⚠️ เรื่อง {เลขที่} ใกล้ครบกำหนด เหลืออีก {N} วัน" |
| N-09 | เกินกำหนด | Officer, Chief, Agency Officer/Head | "🔴 เรื่อง {เลขที่} เกินกำหนดแล้ว {N} วัน" |
| N-10 | ไม่มีการอัพเดตสถานะครบ 30 วัน (เร่งรัดครั้งที่ 1) | เจ้าหน้าที่หน่วยงานผู้รับผิดชอบ (ตนเอง) | "⏰ เรื่อง {เลขที่} ไม่มีการอัพเดตความคืบหน้ามาแล้ว {N} วัน กรุณาอัพเดตสถานะ" |
| N-11 | ไม่มีการอัพเดตครบ 45 วัน (+15 วัน, ครั้งที่ 2) | หัวหน้าหน่วยงาน + เจ้าหน้าที่ผู้รับผิดชอบ | "⏰ เรื่อง {เลขที่} ค้างอัพเดต {N} วัน — แจ้งหัวหน้าหน่วยงานเร่งรัด" |
| N-12 | ไม่มีการอัพเดตครบ 52 วัน (+7 วัน, ครั้งที่ 3) | หัวหน้าหน่วยงาน + เจ้าหน้าที่ผู้รับผิดชอบ | "🔴 เรื่อง {เลขที่} ค้างอัพเดต {N} วัน — โปรดเร่งรัดโดยด่วน" |

> **N-10…N-12 (Status-Update Escalation):** แจ้งเตือนแบบขั้นบันได เมื่อเรื่องที่อยู่ระหว่างดำเนินการ (ACCEPTED/IN_PROGRESS) ไม่มีการอัพเดตความคืบหน้าครบ 30 → 45 → 52 วัน (นับจากการอัพเดตล่าสุด หรือวันรับเรื่องถ้ายังไม่เคยอัพเดต) · มีอัพเดต → รีเซ็ตวงรอบ · เป็นกลไกแยกจาก SLA (ดู `08` §4.3)

| # | การตัดสินใจ | เหตุผล |
|---|-----------|--------|
| KD-1 | OVERDUE เป็น Flag ไม่ใช่สถานะ | เพื่อให้เรื่องยังคงแสดงสถานะจริง (เช่น IN_PROGRESS) และเห็นว่าเกินกำหนดด้วย |
| KD-2 | ประชาชนเห็นสถานะแบบย่อ | ป้องกันความสับสนและปกป้องข้อมูลภายใน |
| KD-3 | CLOSED และ REJECTED เป็น Final Status | ป้องกันการเปลี่ยนแปลงข้อมูลหลังปิดเรื่อง |
| KD-4 | เรื่องส่งคืน (RETURNED) ต้องกลับไป SCREENING | เพื่อให้ศูนย์ดำรงธรรมได้ตรวจสอบอีกครั้งก่อนส่งต่อใหม่ |
| KD-5 | ยื่นเรื่องได้ทั้งสมาชิก/Guest/Staff โดยใช้ Workflow เดียวกัน (เริ่มที่ NEW) | ไม่แยก flow — ต่างกันแค่ source/citizen_id (ดู §5 Step 0) |
| KD-6 | ปกปิดตัวตน (anonymous) ไม่กระทบ Workflow | เป็นเพียงการ mask ข้อมูลผู้ร้อง สถานะ/Transition ทั้งหมดเหมือนเดิม |
