📋 สิ่งที่ต้องมีใน SKILL.md (ทุกข้อจำเป็น):

1. YAML Frontmatter — ด้านบนสุดของไฟล์ อยู่ระหว่าง --- ประกอบด้วย:
   - name: (lowercase, ใช้ - คั่น, ต้องตรงกับชื่อโฟลเดอร์)
   - description: (สั้นกระชับ ไม่เกิน 200 ตัวอักษร ใช้ keyword ที่ agent จะ match ได้)
   - version: (semantic version เช่น 1.0.0)
   - author:
   - tags: (รายการ keyword เช่น docker, react, nodejs)

2. Overview — อธิบายสั้นๆ ว่าโปรเจกต์นี้คืออะไร ทำอะไร

3. When to Use — ระบุชัดเจนว่า skill นี้ควรถูกเรียกใช้เมื่อไหร่
   (เช่น เมื่อทำงานกับ Docker, เขียน API, แก้ไข frontend)

4. When NOT to Use — ระบุขอบเขตที่ skill ไม่ควรถูกเรียกใช้
   (เช่น งานที่ไม่เกี่ยวกับโปรเจกต์นี้, production deployment)

5. Project Architecture — รายละเอียดทางเทคนิค:
   - Directory Structure (โครงสร้างโฟลเดอร์ทั้งหมด)
   - Service Map & Ports (ตาราง: service, container name, host port, container port, URL)
   - Network Communication Rules (service คุยกันยังไง ใช้ hostname อะไร port อะไร)
     ‣ เน้นย้ำ: ภายใน Docker ใช้ชื่อ service + internal port เท่านั้น
     ‣ ห้ามใช้ host-mapped port สำหรับ inter-container communication
   - Environment Variables (ตาราง: ชื่อ, ค่าเริ่มต้น, ใช้โดย service ไหน)

6. Instructions — คำสั่งที่ใช้บ่อย:
   - Docker Commands (start, stop, rebuild, logs, exec)
   - Dependency Management (วิธีเพิ่ม npm/pip package แล้ว rebuild)
     ‣ ระบุให้ชัดว่าเมื่อไหร่ต้อง rebuild vs. แค่ restart
   - Database Management (วิธีรัน migration, เชื่อมต่อ CLI, reset data)

7. Coding Guidelines — แนวทางเขียนโค้ดสำหรับแต่ละ layer:
   - Frontend: component structure, state management, API calls, styling, error handling
   - Backend: project structure, route conventions, database access, error handling, security
   - Database: naming conventions, character set, migration strategy

8. Output Format — บอก agent ว่าเมื่อสร้างโค้ด ควรวางไฟล์ที่ไหน

9. Examples — ตัวอย่าง 2-3 สถานการณ์ พร้อมขั้นตอนที่ agent ควรทำ
   (เช่น 'เพิ่ม API endpoint ใหม่', 'เพิ่มตารางในฐานข้อมูล')

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

⚠️ ข้อกำหนดสำคัญ:

- เขียน description ใน YAML ให้กระชับ เพราะ agent ใช้ตรงนี้ตัดสินใจว่าจะเรียก skill หรือไม่
- เน้น Network Rules ให้ชัดเจน (โดยเฉพาะ hostname และ internal port ของ Docker)
- ใช้ตาราง Markdown สำหรับข้อมูลที่เปรียบเทียบกันได้ (ports, env vars)
- ใส่ code block ตัวอย่างจริงใน Coding Guidelines
- ข้อมูลที่ agent ต้องรู้แน่ๆ ใส่ไว้ใน SKILL.md โดยตรง
- ข้อมูลที่ยาวมาก (เช่น API spec) ให้แยกไปไว้ในโฟลเดอร์ references/ แล้วอ้างอิง
- ความยาวรวมของ SKILL.md ไม่ควรเกิน 5,000 tokens
- อย่าใส่ version ใน docker-compose.yml (ล้าสมัยแล้ว)"

================================================================================
  💡 เคล็ดลับเพิ่มเติม
================================================================================

1. ชื่อ skill (name) ต้องตรงกับชื่อโฟลเดอร์เป๊ะๆ
   - โฟลเดอร์: .agents/skills/my-project-dev/
   - name ใน YAML: my-project-dev

2. description เป็นตัวชี้ขาดว่า agent จะเรียกใช้ skill หรือไม่
   - ✅ ดี: "Manages MyShop e-commerce app. Use for Docker, Next.js frontend,
          Django backend, PostgreSQL."
   - ❌ ไม่ดี: "A skill for my project."

3. When NOT to Use สำคัญมาก — ป้องกัน agent เรียกใช้ skill ผิดจังหวะ

4. ถ้าโปรเจกต์ใหญ่มาก ให้แยกเป็นหลาย skills ได้ เช่น:
   - .agents/skills/myapp-docker/SKILL.md    (เรื่อง Docker เท่านั้น)
   - .agents/skills/myapp-frontend/SKILL.md  (เรื่อง Frontend เท่านั้น)
   - .agents/skills/myapp-backend/SKILL.md   (เรื่อง Backend เท่านั้น)

5. อัปเดต SKILL.md เป็นประจำเมื่อโปรเจกต์เปลี่ยนแปลง
   เช่น เพิ่ม service ใหม่, เปลี่ยน port, เพิ่ม convention ใหม่

6. ตรวจสอบ port ที่ OS จอง ก่อนกำหนดใน Docker:
   - macOS: lsof -i :<port>
   - Linux: ss -tlnp | grep <port>

7. ถ้าต้องการให้ agent รู้จัก API schema หรือ DB schema ที่ซับซ้อน
   ให้แยกเป็นไฟล์ใน references/ แล้วอ้างอิงใน SKILL.md เช่น:
   "ดูรายละเอียด API ได้ที่ references/api-spec.md"
